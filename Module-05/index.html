Passing By: Value VS. Reference

In programming, there 2 fundamental concepts of how data is passed around a program: By Value and By Reference. It is important to understand this concept as it helps you understand how prototypal inheritance works.

Pass By Value

By value means the value of what you're assigning is copied into a new address location. Everytime you assign x to another variable it will continually copy the value in the memory location into the new allocated memory location. Changing the original x will have no affect on any of the other values.

Example: Pass by Value

In the image above you can see we've set x and made the value equal to 5. We've set y to be the value of x. As you will notice, the value is moved into a new memory location disconnecting it from x entirely. As you can see, logging x and y shows the values are completely different. This is because in JavaScript, values are passed by value (copied) and not by reference.

Pass By Reference

As we keep saying "Most things in JavaScript are objects". Objects by their very nature are passed around by reference. This means that instead of creating a complete duplicate of the object only the memory address of the object is passed. So, when you attempt to access any properties or methods, you're always accessing the original. The largest benefit to that is that any modifications to the original will obviously be available to any of the references.

Example: Pass by Reference

In the image above we are creating a new variable, x and assigning it the value of an object with the property a. a has the value of 5. Next we create a new variable, y and we assign x to it. When we change the value of x.a it changes the value stored in the address location. Every variable currently referencing x has access to the property a and therefore the updated value.

Inheritance Using Object Constructors

Recap: Creating an Object Constructor

Object Constructors are simply just a standard function. We can use these to create a blueprint (just like classes do) for what we want an object to look like -- any necessary properties it may need to contain. The one thing we haven't really discussed is what happens when we create an object using the object constructor method.

The new keyword is what makes creating objects with object constructors doable. new does the following:

  1. It creates a new empty object
  2. It sets a new property on the object called prototype
  3. It binds all the properties set with the this keyword to the object
  4. It returns the newly created object

So that's it, in a nutshell. Simply put, it translates our function into an object.

Prototype

Sounds like a robot horror movie where the robot becomes sentient and murders all the harmless scientists! But it isn't. Prototype is a property automatically given to functions. When you define a new object constructor you can also define the properties and methods any children objects can inherit by adding them to the prototype bucket. Any properties defined inside the constructor are immediately added. External ones would have to be added by first accessing the prototype property then attaching the new property to it. In short, then, a prototype is a property on a function that points to an object.

Inheritance Using Object Prototypes

We can have one object constructor inherit from another object constructor. As discussed, an individual Person is likely to have more traits then afforded by the Person object. If we want to give it more traits using inheritance, we will need another object constructor. Why don't we create the Superhero object constructor.

In order to inherit from our Person object, we must instantiate it using the .call() method. This is essentially like calling super() in other languages as it will instantiate the Person object and assign its properties to SuperHero where they are not already defined.

The call method basically allows you to call a function and then pass in a specific value for the this keyword binding the properties of that function to the this owner.

Example: Creating a new object 
// Employee inheriting from Person
function Employee (name, age) {
  // This refers to the Employee owner
  Person.call(this, name, age);
}

In order to use all methods assigned to the Person prototype property, we must set a reference to it on our SuperHero constructor. Currently we have access to only the properties defined in the original Person structure, so we don't have access to any methods defined on the Person.prototype.

Example: Creating an inherited object 
// Employee inheriting from Person
function Employee (name, age) {
  // When you attempt a method on Person that doesn't exist on Employee it will move up the chain. That's why we're passing 'this' to Person.call(). It will attempt to call the method on Employee first, then move to calling it on Person as a fallback
  Person.call(this, name, age);
}

// This will assign the Person.prototype references (properties and other inheritance) to the Employee prototype property. This will make it so any prototypal properties we've assigned to Person will now be available to Employee
Employee.prototype = Object.create(Person.prototype);

Composition Over inheritance

Composition Over Inheritance - Fun Fun Function

Multiple inheritance isn't supported in every language, JavaScript being one of them. The idea of multiple inheritance is that an object can inherit properties from multiple objects at the same ancestry level. Unfortunately that is not the case as JavaScript requires chain inheritance where one object must be a child of another, but will pass the parent's properties as long as they haven't been overwritten. That is why many JavaScript programmers prefer to program with composition over inheritance. Composition, simply put, is creating an object that has the exact traits it requires. For example, a person can move and make noise by talking, however a dog can also move and make noise by barking. The implementation of noise is drastically different as the human has the ability to make millions of noises that form words and sounds where as the dog only has a few. So well both human and dog could inherit from a parent class like mammal the implementation would require both creatures to implement a different noise method. This is where composition makes more sense over inheritance.

With composition we can create a trait called talk and a trait called bark. We can also create a trait called walk. The human will have the traits talk and walk; the dog will have the traits bark and walk. Our code would still be nicely encapsulated and modular, but we would be able to efficiently give each creature its specified properties.

To sum up the difference: Inheritance is when you design your types around what they are. Composition is when you design your types around what they do.
